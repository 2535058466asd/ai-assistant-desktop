# AI Agent / 个人助手开源项目调研与 Nova 对比

> 调研日期：2026-05-13  
> 调研目标：多看同类和邻近项目，判断 Nova 个人 AI 助手的优势、短板，以及后续应该借鉴哪些成熟设计。  
> 注意：GitHub star 数和项目描述会变化，本文只作为当前阶段的技术方向参考。

## 1. 先给结论

Nova 不是最强的“桌面控制 Agent”，也不是最成熟的“企业级 Agent 平台”，但它有一个比较清晰的作品集定位：

**本地个人 AI 工作台 / 个人助手桌面端。**

它的价值不在于单点能力碾压 UI-TARS、Dify、Open WebUI 这些大项目，而在于把下面几件事组合在一个 Electron 桌面应用里：

- Chat UI
- Function Calling Agent Loop
- 本地工具调用
- 本地文件读写
- Web 搜索
- RAG 知识库
- 长期记忆
- TTS / ASR 语音交互
- 多模型 Provider
- 工具日志和可观测性

这个组合比“单纯套模型聊天壳”更完整，也比“只做爬虫 / 只做自动化 / 只做编程助手”更接近一个真正的个人 AI 应用。

当前最应该学习的不是“全部迁移成某个框架”，而是：

1. 从 UI-TARS / Agent TARS 学习 **过程展示、事件流、屏幕理解和工具执行反馈**。
2. 从 AnythingLLM / Open WebUI / Dify 学习 **设置页、Provider 管理、RAG 来源展示、权限和多用户结构**。
3. 从 Open Interpreter 学习 **命令执行确认、安全边界、verbose 调试模式**。
4. 从 LangGraph / AutoGen 学习 **状态机、人机确认、多 Agent 编排思想**，但第一阶段不必重构进去。
5. 从 Vercel AI SDK 学习 **统一 Provider、流式事件、tool call 事件设计**。

## 2. 项目分组

| 类型 | 项目 | 和 Nova 的关系 |
| --- | --- | --- |
| 桌面 / 电脑控制 Agent | UI-TARS Desktop / Agent TARS、Open Interpreter、Jan | 借鉴桌面操作、权限确认、本地运行体验 |
| RAG / 知识库 / 聊天平台 | AnythingLLM、Open WebUI、Dify、Khoj | 借鉴知识库、模型配置、来源引用、Agent 设置 |
| Agent 框架 | LangGraph、AutoGen、Vercel AI SDK | 借鉴状态流、事件流、Provider、工具调用协议 |
| Agent 工作空间 / 多 Agent | LobeHub | 借鉴 Agent 人格、Agent 工作区、记忆可编辑 |
| 编程 Agent / 工程增强 | Continue | 借鉴可观测性、仓库级配置、检查任务，但不是 Nova 主线 |

## 3. 重点项目分析

### 3.1 UI-TARS Desktop / Agent TARS

链接：[bytedance/UI-TARS-desktop](https://github.com/bytedance/UI-TARS-desktop)

定位：多模态 GUI Agent / 浏览器和电脑操作 Agent。README 里明确强调 GUI Agent、Vision、MCP 工具、Event Stream、浏览器和电脑操作。

优点：

- 屏幕视觉理解和 GUI 自动化能力强。
- 支持浏览器操作、电脑操作、MCP 工具接入。
- 有实时反馈和状态展示，对 Agent 执行过程非常重视。
- 官方项目结构成熟，有 CLI、Web UI、Desktop 多入口。

缺点 / 不适合直接照搬的点：

- 重点是“控制电脑”，不是个人长期助手。
- 对视觉模型和操作安全要求高，复杂度明显高于 Nova 当前阶段。
- 如果 Nova 现在强行做连续键鼠控制，会拉高风险和维护成本。

Nova 可借鉴：

- 工具执行过程要可视化：正在看屏幕、正在点击、正在等待、失败原因。
- 截图理解可以作为轻量能力接入，不要一开始就做完整自动键鼠。
- Event Stream Viewer 思路值得学：用户应该看到 Agent 每一步做了什么。

### 3.2 Open Interpreter

链接：[openinterpreter/open-interpreter](https://github.com/openinterpreter/open-interpreter)

定位：给电脑提供自然语言接口，可以让 LLM 在本地运行 Python、JavaScript、Shell 等代码。README 强调会在运行代码前询问用户确认。

优点：

- “自然语言控制本地电脑”的定位非常清楚。
- 命令执行和代码执行能力强。
- 有确认机制、verbose 模式、profile 配置。
- 支持 OpenAI-compatible / 本地模型，Provider 思路成熟。

缺点：

- 偏 CLI 和代码执行，不是完整桌面个人助手。
- 高权限执行风险大，不适合默认全自动。
- RAG、记忆、桌面 UI 不是它的主优势。

Nova 可借鉴：

- `exec_command` 这类系统工具要分级：日常可自动，高危必须确认。
- 做一个“详细调试模式”：显示模型请求、工具调用、结果摘要、失败栈。
- 支持 profile / preset，例如“日常模式”“开发模式”“安全模式”。

### 3.3 AnythingLLM

链接：[Mintplex-Labs/anything-llm](https://github.com/Mintplex-Labs/anything-llm)

定位：本地优先的 all-in-one AI 应用，支持文档聊天、Agent、多用户、多模型、向量数据库、文档管道、桌面版。

优点：

- RAG 产品化程度高：文档导入、来源引用、文档管道比较完整。
- 支持很多 LLM、Embedding、Vector DB、TTS/STT。
- 桌面版、Docker、自托管都有。
- “Intelligent Skill Selection” 很值得关注：通过智能选择工具降低每次请求携带的工具 token。

缺点：

- 体量大，复杂度远高于个人项目。
- 更像通用知识库 / 团队平台，不是轻量个人桌面助手。
- 代码和部署链路较重，不适合 Nova 当前直接模仿。

Nova 可借鉴：

- 工具太多以后，不要每次都把所有工具塞给模型。可以做工具候选选择。
- RAG 结果要稳定展示来源、文件名、chunk、时间、分类。
- 设置页可以分为模型、知识库、语音、工具、安全几个大区。

### 3.4 Open WebUI

链接：[open-webui/open-webui](https://github.com/open-webui/open-webui)

定位：自托管 AI 平台，支持 Ollama、OpenAI-compatible API、离线运行、RAG、权限、工具、语音等。

优点：

- UI 成熟，模型管理、用户权限、RAG、语音、工具生态都比较全。
- 支持 OpenAI-compatible API，模型接入非常灵活。
- 有本地 RAG、文档库、工具工作区等成熟产品表达。
- 社区大，功能覆盖广。

缺点：

- 是 Web 平台，不是 Electron 本地桌面产品。
- 面向多人和部署，很多复杂能力对 Nova 当前阶段是负担。
- 后端依赖和部署方式比 Nova 重。

Nova 可借鉴：

- 模型 Provider 设置界面。
- RAG 文档库和来源引用 UI。
- 工具工作区 / Function 管理 UI。
- 权限分级和用户可见的开关。

### 3.5 Dify

链接：[langgenius/dify](https://github.com/langgenius/dify)

定位：生产级 LLM 应用开发平台，组合了 Workflow、RAG Pipeline、Agent、模型管理、可观测性、API。

优点：

- Workflow 很成熟，适合企业应用编排。
- 模型管理、RAG、Agent、工具、日志、评估都完整。
- 从原型到生产的路径清晰。
- 可观测性和应用日志做得比个人项目强很多。

缺点：

- 不是桌面应用，也不是个人助手。
- 平台思维重，开发和部署成本高。
- 对 Nova 来说，照搬 Workflow 画布会让项目过早变复杂。

Nova 可借鉴：

- 可观测性：每次对话、模型调用、工具调用、RAG 命中都应该能追踪。
- Eval 面板方向是对的，可以继续做成固定演示集。
- 后续如果做“办公流程自动化”，可以借鉴 Workflow，但不要第一阶段就做画布。

### 3.6 Jan

链接：[janhq/jan](https://github.com/janhq/jan)

定位：开源 ChatGPT 替代品，强调本地运行、隐私、下载和运行本地模型，使用 Tauri。

优点：

- 本地模型体验非常清晰，隐私定位强。
- 跨平台桌面产品成熟。
- 支持 OpenAI-compatible API，本地 server 可以给其他应用调用。
- 轻量、产品化、用户安装路径清晰。

缺点：

- 更像本地模型聊天客户端，不是工具型 Agent。
- RAG、复杂工具调用、个人任务执行不是它的核心。

Nova 可借鉴：

- 本地优先和隐私优先的产品表达。
- 模型下载 / 模型服务检测 / 本地 API Server 的 UI。
- 如果后续接 Ollama / LM Studio，Jan 是很好的参考对象。

### 3.7 Khoj

链接：[khoj-ai/khoj](https://github.com/khoj-ai/khoj)

定位：AI second brain，自托管，能从网页和文档回答，支持自定义 Agent、定时自动化、深度研究、多入口使用。

优点：

- “第二大脑”定位和 Nova 的长期记忆方向接近。
- 支持文档、网页、自定义 Agent、自动化、通知。
- 可从 Browser、Obsidian、Emacs、Desktop、Phone、Whatsapp 等入口访问。

缺点：

- 是平台 / 服务形态，不是单纯桌面应用。
- 功能宽，部署和使用路径比 Nova 重。

Nova 可借鉴：

- 记忆系统要可解释、可查看、可编辑。
- 自动化不一定要复杂，先从“定时总结、提醒、整理资料”做起。
- 多入口方向有价值，但当前阶段先把桌面端做好。

### 3.8 LobeHub

链接：[lobehub/lobehub](https://github.com/lobehub/lobehub)

定位：Agent 工作空间，把 Agent 当成工作单元，支持 Agent Builder、多 Agent 协作、MCP、记忆、知识库、多模型、语音、插件系统等。

优点：

- Agent 作为“工作伙伴”的产品叙事很强。
- 人格、技能、记忆、工作区、团队协作的概念完整。
- MCP 插件、Agent Market、多模型支持都值得参考。
- 明确提到结构化、可编辑的记忆。

缺点：

- 愿景很大，复杂度高。
- 很多能力偏平台化和多人协作，对 Nova 当前阶段不是刚需。

Nova 可借鉴：

- “启源 AI / Nova”的人格和设置不要只是写死在 prompt 里，应该产品化。
- 记忆应该白盒化：用户能看到、编辑、删除、禁用。
- 后续可以有多个 Agent 角色，但第一阶段不要急着做多 Agent 协作。

### 3.9 LangGraph

链接：[langchain-ai/langgraph](https://github.com/langchain-ai/langgraph)

定位：构建长时间运行、有状态 Agent 的底层编排框架。README 强调 durable execution、human-in-the-loop、memory、debugging。

优点：

- 状态图思想成熟，适合复杂 Agent。
- 人机确认、持久化执行、长期记忆、调试链路都有明确设计。
- 对“工具执行流程”和“中断恢复”很有参考价值。

缺点：

- 框架重，迁移成本高。
- 如果 Nova 现在全面迁移，会打乱已有 Function Calling Loop。
- 对前端桌面体验帮助有限，主要是后端编排思想。

Nova 可借鉴：

- 不迁移框架，但吸收状态机思想。
- 把 Agent Loop 的状态明确化：用户输入、模型思考、工具请求、确认、执行、总结、结束。
- 高风险工具走 human-in-the-loop。

### 3.10 AutoGen / Microsoft Agent Framework

链接：[microsoft/autogen](https://github.com/microsoft/autogen)

定位：多 Agent AI 应用框架。当前 AutoGen README 提到项目进入维护模式，建议新用户转向 Microsoft Agent Framework。

优点：

- 多 Agent 协作思想成熟。
- AgentChat、工具、MCP、Studio 等方向值得了解。
- 适合复杂协同任务。

缺点：

- AutoGen 本体已进入维护模式，不适合作为新项目核心依赖。
- 多 Agent 会显著提高复杂度，Nova 当前还没必要。

Nova 可借鉴：

- 只借鉴概念：任务型 Agent、工具型 Agent、审查型 Agent。
- 不建议现在引入完整多 Agent 框架。

### 3.11 Vercel AI SDK

链接：[vercel/ai](https://github.com/vercel/ai)

定位：TypeScript AI 应用和 Agent 工具包，强调 provider-agnostic、统一 API、streaming、structured output、tools / agents。

优点：

- TypeScript 技术栈和 Nova 接近。
- Provider 抽象、流式输出、结构化输出、工具调用设计值得重点参考。
- API 设计简洁，适合作为 Nova 自研 Provider 的参考。

缺点：

- 更偏 Web / Next.js 生态，不是 Electron 桌面应用方案。
- 直接引入可能会和现有 Provider 抽象重复。

Nova 可借鉴：

- 当前统一模型配置方向是对的。
- 后续流式事件建议标准化：`text_delta`、`tool_call_delta`、`tool_started`、`tool_finished`、`error`。
- 工具 schema 和执行逻辑继续保持单一来源。

### 3.12 Continue

链接：[continuedev/continue](https://github.com/continuedev/continue)

定位：现在更偏“代码仓库 AI 检查 / CI 中运行的 Agent”，也有 VS Code / JetBrains 生态。

优点：

- 仓库级配置、规则文件、自动检查的产品思路好。
- 对开发者工作流理解深。
- 可观测、可复现、可配置。

缺点：

- 不是通用个人助手。
- 和 Nova 的“办公 / 个人桌面助手”主线不完全一致。

Nova 可借鉴：

- 可以给项目加 `eval`、`demo`、`agent rules` 这类可复现配置。
- 不建议把 Nova 变成代码助手。

## 5. Nova 目前真正有价值的差异

### 5.1 比“普通聊天壳”强

Nova 已经不是简单的模型聊天 UI，因为它有：

- Function Calling Agent Loop
- 本地文件 / 剪贴板 / 截图 / 搜索 / 知识库工具
- 记忆提取
- 工具日志
- 多模型 Provider
- 语音链路

这些是 AI 应用岗位里很有说服力的能力点。

### 5.2 比“重型平台”轻

Dify、Open WebUI、AnythingLLM 很强，但它们是平台。Nova 的优势是：

- 本地桌面体验更直接。
- 适合个人使用，不需要 Docker / 服务部署。
- 可以更快迭代个人助手体验。
- 更适合作为作品集讲清楚“我自己实现了 Agent Loop、工具系统、RAG 和记忆”。

### 5.3 比“纯自动化 Agent”更安全

UI-TARS 和 Open Interpreter 的能力强，但高权限风险也高。Nova 现在的克制反而是优点：

- 可以先做读文件、搜索、总结、知识库、轻量系统操作。
- 高风险命令弹窗确认。
- 不急着做连续键鼠控制，避免演示翻车。

### 5.4 比“单纯 RAG 项目”更像个人助手

AnythingLLM / Open WebUI 的 RAG 很强，但 Nova 还有个人记忆、语音和桌面工具。这让它更像“我的 AI 工作台”，而不是“文档问答系统”。

## 6. Nova 目前的短板

### 6.1 产品边界还需要收敛

现在 Nova 同时想做：

- 聊天助手
- RAG
- 记忆
- 语音
- 桌面工具
- 工作区任务
- Eval
- 设置面板

这没问题，但需要一个明确主线：

**主线建议：个人 AI 工作台，不是代码助手，不是全自动电脑控制，不是企业工作流平台。**

### 6.2 可观测性还不够产品化

控制台日志已经比以前好，但用户界面里的过程展示还可以继续强化：

- 模型正在分析
- 选择了哪个工具
- 为什么调用这个工具
- 工具参数摘要
- 工具执行耗时
- 工具失败原因
- 最终根据哪个结果回答

注意：真正的“模型隐藏推理链”多数 API 不会完整暴露，也不应该强求展示。更合理的是展示 **可解释的执行过程**，而不是泄露模型内部 chain-of-thought。

### 6.3 工具太多后会增加 token 成本

AnythingLLM 的 “Intelligent Skill Selection” 思路值得借鉴。Nova 后续可以做两层工具选择：

1. 先用轻量规则或小模型选出候选工具。
2. 再把候选工具 schema 给主模型。

这样避免每次聊天都塞完整工具列表。

### 6.4 记忆系统要白盒化

长期记忆如果用户看不到、改不了，会变成黑盒风险。建议后续做：

- 记忆列表
- 记忆分类
- 重要性
- 来源对话
- 禁用 / 删除
- 手动添加
- 错误记忆修正

### 6.5 设置页需要重新组织

当前设置页如果继续塞功能，会越来越窄。建议改为：

- 模型：Provider、API Key、Base URL、默认模型、压缩模型
- 语音：ASR、TTS、音色、试听
- 知识库：向量库、导入、来源、Eval
- 工具：启用/禁用、风险等级、确认策略
- 记忆：记忆开关、自动提取、列表管理
- 安全：命令权限、外发消息权限、文件写入权限
- 调试：详细日志、模型请求、工具事件、导出日志

## 7. 建议吸收的设计清单

### 近期可以做

1. Agent 过程面板
   - 来源：UI-TARS / Agent TARS、Dify、LangGraph
   - 做法：展示执行状态，不展示隐藏推理链。

2. 工具候选选择
   - 来源：AnythingLLM Intelligent Skill Selection
   - 做法：按关键词、上下文、工具风险等级过滤工具。

3. 记忆管理页
   - 来源：LobeHub、Khoj
   - 做法：可查看、编辑、删除、禁用。

4. Provider 设置页升级
   - 来源：Open WebUI、Jan、Vercel AI SDK
   - 做法：统一配置、测试连接、模型列表、默认模型。

5. 安全模式 / 开发模式
   - 来源：Open Interpreter
   - 做法：普通用户默认安全，开发时可打开详细日志。

### 中期再做

1. 截图理解工具
   - 不做连续键鼠，只做“看屏幕并解释 / 给建议”。

2. RAG 来源浏览和删除
   - 按文件、chunk、时间、分类管理知识库。

3. Eval 固定演示集
   - 用来证明 RAG、记忆、工具调用确实可用。

4. 自动化任务
   - 先做低风险任务：每日总结、资料整理、提醒、读取文件生成摘要。

### 暂时不建议做

1. 完整 Workflow 画布
   - 太重，容易偏离个人助手主线。

2. 完整多 Agent 协作
   - 可以先有概念，不要马上工程化。

3. 连续键鼠桌面控制
   - 风险高，演示不稳定，先做截图理解和单步工具。

4. 全面迁移 LangChain / LangGraph
   - 当前自研 Agent Loop 已经能工作，重构收益不够高。

## 8. Nova 后续路线建议

### 阶段 1：作品集稳定版

目标：让项目“能讲清楚、能演示、能构建、能复现”。

- 清理死代码和断链。
- 保持 build 通过。
- README 写清楚架构。
- 做 3 个固定 Demo：
  - 导入文档后问答，并展示来源。
  - 读取本地文件，总结并写入任务。
  - 截图 / 剪贴板分析，并给建议。
- Agent 过程面板显示工具调用链。

### 阶段 2：个人助手增强版

目标：从“能用”变成“像产品”。

- 设置页重构。
- 记忆管理页。
- Provider 测试连接。
- RAG 来源管理。
- 工具启用 / 禁用 / 权限策略。

### 阶段 3：高级 Agent 能力

目标：形成差异化。

- 截图理解。
- 工具候选选择，降低 token 成本。
- 自动化任务。
- 可选 MCP 接入。
- 可选本地模型服务检测。

## 9. 最终判断

Nova 的方向是成立的，但不要和 Dify / Open WebUI / UI-TARS 这些大项目硬拼全能力。

更好的定位是：

> 一个面向个人使用的本地 AI 工作台，重点展示 Agent Loop、工具调用、RAG、记忆、多模型和桌面能力。

这个定位比“套壳聊天机器人”更高级，也比“做一个巨大平台”更适合个人作品集。

如果后续要面试或展示，建议重点讲：

1. 我没有直接套 LangChain，而是自己实现了 Function Calling Agent Loop。
2. 我做了 Model Provider 抽象，可以接豆包、MiMo、OpenAI-compatible。
3. 我做了 Tool Registry，并给工具加风险等级和确认策略。
4. 我做了 RAG 知识库和长期记忆，能处理个人上下文。
5. 我做了 Electron 桌面端，能调用本地文件、剪贴板、截图、系统命令。
6. 我知道安全边界在哪里，所以没有盲目做高危连续桌控。

这套说法是有技术含量的，也比较诚实。
