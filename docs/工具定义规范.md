# 启源 AI - 工具定义规范

> 参考豆包官方文档：https://www.volcengine.com/docs/82379/1262342

## 一、工具定义三要素（缺一不可）

每个工具必须包含 `name`、`description`、`parameters`：

```json
{
  "type": "function",
  "function": {
    "name": "get_weather",
    "description": "获取指定地点的天气信息，支持摄氏度和华氏度两种单位",
    "parameters": {
      "type": "object",
      "properties": {
        "location": {
          "type": "string",
          "description": "地点的位置信息，例如北京、上海"
        },
        "unit": {
          "type": "string",
          "enum": ["摄氏度", "华氏度"],
          "description": "温度单位，可选值为摄氏度或华氏度"
        }
      },
      "required": ["location"]
    }
  }
}
```

## 二、description 写法规范

### 核心原则：让 AI 准确判断"什么时候该用这个工具"

| 规则 | 好的写法 | 差的写法 |
|------|---------|---------|
| 说清楚干什么 | "列出目录中的文件和文件夹" | "获取文件系统信息" |
| 说清楚适用场景 | "当用户想查看某个文件夹里有什么时使用" | （没有） |
| 给具体例子 | "路径示例：~/Desktop（桌面）、C:\\Users\\xxx\\project" | "传入一个路径" |
| 有固定取值用 enum | `"enum": ["摄氏度", "华氏度"]` | "传入温度单位" |

### description 模板

```
[工具功能描述]。[适用场景描述]。[参数示例/取值说明]。
```

示例：
```
"列出目录中的文件和文件夹。当用户想查看某个文件夹里有什么时使用。
路径示例：~/Desktop（桌面）、~/Documents（文档）、~/Downloads（下载），
或任意绝对路径如 C:\\Users\\xxx\\project。"
```

## 三、parameters 写法规范

### 1. 每个参数都要有 description

```json
{
  "path": {
    "type": "string",
    "description": "文件或目录的路径，例如 ~/Desktop/天气.txt"
  }
}
```

### 2. 有固定取值的参数用 enum

```json
{
  "unit": {
    "type": "string",
    "enum": ["摄氏度", "华氏度"],
    "description": "温度单位"
  }
}
```

enum 的作用：限制 AI 的填值范围，防止参数幻觉。

### 3. 必填参数放 required

```json
{
  "required": ["path"]
}
```

不在 required 里的参数，AI 可以选择不填。

### 4. 参数类型要准确

| 场景 | type | 说明 |
|------|------|------|
| 文本 | "string" | 最常用 |
| 布尔 | "boolean" | 开关类参数 |
| 数字 | "number" / "integer" | 数量、次数等 |

## 四、中间层转换规范

AI 传的参数 → 中间层转换 → 底层 Node.js API

### 核心原则：告诉 AI "能传什么"，而不是"底层要什么"

AI 不需要知道底层用的是 `fs.readdirSync()` 还是 `child_process.exec()`。

### 路径转换

AI 可能传的路径格式：

| 格式 | 例子 | Node.js 认识？ | 处理方式 |
|------|------|---------------|---------|
| 绝对路径 | `C:\Users\xxx\Desktop` | ✅ | 直接用 |
| 相对路径 | `./Desktop` | ✅ | 直接用 |
| `~` 路径 | `~/Desktop` | ❌ | `resolvePath()` 转换 |

**所有涉及路径的工具都必须调用 `resolvePath()`**，包括：
- `read_file` ✅
- `write_file` ✅
- `list_dir` ✅
- `search_files` ✅
- `grep_content` ✅

### 错误处理

底层返回的错误要转成 AI 能理解的格式：

```json
{ "success": false, "error": "文件不存在: ~/Desktop/不存在的文件.txt" }
```

## 五、豆包内置工具

除了自定义 Function Calling，豆包还提供内置工具，可以直接在 `toolDefinitions` 数组中添加：

| 工具 | 添加方式 | 说明 |
|------|---------|------|
| **联网搜索** | `{ type: "web_search" }` | 豆包自动联网搜索，无需自己写搜索代码 |
| 豆包助手 | `{ type: "doubao_app", ... }` | 需要 `ark-beta-doubao-app` header |
| 图像处理 | `{ type: "image_process" }` | 需要视觉模型 + `ark-beta-image-process` header |
| 知识库搜索 | `{ type: "knowledge_search", ... }` | 需要旗舰版知识库 + `ark-beta-knowledge-search` header |

### 内置工具在 orchestrator 中的处理

内置工具没有 `function` 字段，由豆包服务端自行处理，不需要我们执行：

```typescript
for (const toolCall of message.tool_calls) {
  if (toolCall.type === 'web_search' || !toolCall.function) {
    // 内置工具，跳过，豆包会在下一轮响应中返回结果
    continue;
  }
  // 自定义工具，正常执行
  const toolName = toolCall.function.name;
  // ...
}
```

## 六、添加新工具的 Checklist

添加新工具时，按以下步骤检查：

- [ ] **toolDefinitions.ts**：添加工具的 JSON Schema 定义
  - [ ] name：简洁、唯一
  - [ ] description：说清楚功能 + 适用场景 + 参数示例
  - [ ] parameters：每个参数有 description，有固定取值用 enum，必填放 required

- [ ] **toolExecutor.ts**：添加工具的执行逻辑
  - [ ] 路径参数调用 `resolvePath()`
  - [ ] 返回 `{ success: true, data: ... }` 或 `{ success: false, error: ... }`

- [ ] **fileOps.ts / 其他工具文件**：添加 IPC handler
  - [ ] 参数校验
  - [ ] 错误处理

- [ ] **orchestrator.ts**：如果需要特殊处理（如内置工具），添加判断逻辑

- [ ] **测试**：用自然语言测试 AI 是否能正确选择工具、填对参数

## 七、常见问题

| 问题 | 原因 | 解决方案 |
|------|------|---------|
| AI 不调用工具 | 工具描述不清晰 | 优化 description，添加"适用于…" |
| AI 填错参数 | 参数描述不准确 | 细化 properties 描述，使用 enum |
| AI 填了不存在的路径 | `~` 路径未转换 | 确保调用了 `resolvePath()` |
| 工具结果未被 AI 理解 | 返回格式不规范 | 统一返回 `{ success, data/error }` JSON |
| 同一个工具被调用两次 | 消息添加重复 | 检查 `setMessages` 和 `onMessageCallback` 是否重复 |
